# LiuMa-engine 系统设计说明书

## 项目概述

### 系统定位与核心功能

LiuMa-engine 是一款面向多端（API、WEB、APP）自动化测试的底层执行引擎，作为测试平台的核心组件，负责测试任务的调度、用例执行、结果汇报等关键流程。该引擎采用 Python 语言开发，基于 unittest 测试框架构建，具备高并发、高可靠性和强扩展性的特点。

**核心功能包括：**

- **多任务并发调度**：支持多线程/多进程任务调度，自动拉取平台分配的测试任务
- **多端测试支持**：统一支持 API、WEB、APP 三种类型的自动化测试用例执行
- **智能断言机制**：内置 Assert 断言模块，支持 30+ 种断言类型和多种数据类型校验
- **实时结果汇报**：测试结果实时上传至平台，支持失败重试和批量上传
- **资源自动管理**：自动管理测试过程中的截图、日志等资源，支持自动上传和清理
- **心跳监控机制**：定时发送心跳包维持与平台连接，支持远程任务控制

### 技术架构概述

系统采用分层架构设计，主要分为以下几个层次：

**1. 启动控制层（startup.py）**

- 系统入口点，负责引擎版本管理和启动流程控制
- 调用 Start 模块启动核心服务

**2. 核心调度层（ 模块）**

- **Start**：主调度器，负责心跳连接、任务拉取、消息监听和进程管理
- **Setting**：任务设置器，负责数据拉取、任务解析和线程池管理
- **Run**：执行控制器，负责测试套件构建和用例执行调度
- **Case**：用例容器，基于 unittest.TestCase 扩展，提供事务管理和日志记录

**3. 测试执行层（core 模块）**

- **API 测试模块**：基于 requests 库实现 HTTP 接口测试
- **WEB 测试模块**：基于 Selenium 实现 Web 自动化测试
- **APP 测试模块**：支持移动端应用自动化测试
- **断言模块**：Assert 提供统一的断言验证机制

**4. 支撑服务层（ 辅助模块）**

- **Api**：平台接口通信，负责任务获取、结果上传、文件下载等
- **Report**：结果汇报服务，负责测试结果的收集和批量上传
- **Upload**：资源上传服务，负责截图和文件的自动上传
- **Config**：配置管理服务，负责系统配置的读取和管理
- **Log**：日志服务，提供调试日志和错误日志记录

**5. 工具库层（tools 模块）**

- **utils**：通用工具函数库
- **funclib**：功能函数库，包含数据生成和参数处理等功能

## 业务逻辑设计

### 核心业务流程

#### 1. 引擎启动流程

```
系统启动 → 初始化配置 → 启动心跳连接 → 启动任务拉取 → 启动消息监听 → 进入任务执行循环
```

**详细步骤：**

1. **系统初始化**：加载配置文件，初始化 API 客户端和配置管理器
2. **心跳连接**：通过 WebSocket 与平台建立心跳连接，每 30 秒发送一次心跳包
3. **任务拉取**：定期调用平台接口获取待执行的测试任务
4. **消息监听**：监听平台下发的控制指令（启动、停止、全部停止）
5. **任务调度**：根据最大并发数限制，启动独立进程执行测试任务

#### 2. 任务执行流程

```
接收任务 → 数据拉取 → 任务解析 → 线程池执行 → 结果收集 → 资源上传 → 任务完成
```

**详细步骤：**

1. **任务接收**：从任务队列获取测试任务，包含任务 ID、测试集合、用例列表等信息
2. **数据拉取**：下载任务相关的测试数据文件（ZIP 格式），解压到本地数据目录
3. **任务解析**：解析测试计划，按测试集合分组，为每个用例创建执行上下文
4. **线程池执行**：根据任务配置的最大线程数，使用线程池并发执行测试用例
5. **结果收集**：实时收集测试结果，包括执行状态、日志信息、截图等
6. **资源上传**：自动上传测试过程中产生的截图和日志文件
7. **任务完成**：通知平台任务执行完成，清理本地测试数据

#### 3. 用例执行流程

```
用例初始化 → 前置处理 → 步骤执行 → 断言验证 → 参数提取 → 后置处理 → 结果记录
```

**详细步骤：**

1. **用例初始化**：创建测试用例实例，初始化会话、驱动和上下文
2. **前置处理**：执行前置脚本和 SQL，进行测试环境准备
3. **步骤执行**：根据用例类型（API/WEB/APP）执行相应的测试步骤
4. **断言验证**：使用 Assert 模块进行结果断言验证
5. **参数提取**：从响应结果中提取参数，用于后续用例的数据关联
6. **后置处理**：执行后置脚本和 SQL，进行数据清理和环境恢复
7. **结果记录**：记录执行结果、日志信息和截图，更新事务状态

### 关键业务规则

#### 1. 任务调度规则

- **并发控制**：同时执行的任务数不能超过配置的最大并发数（max-run）
- **任务优先级**：按照任务接收顺序进行调度，先进先出（FIFO）原则
- **失败重试**：支持失败用例的自动重试执行，重试次数由任务配置决定
- **资源隔离**：每个任务使用独立的进程空间，避免相互影响

#### 2. 用例执行规则

- **事务管理**：每个 API 接口或操作步骤作为一个独立事务进行管理
- **错误处理**：支持错误后继续执行（errorContinue）和错误后停止执行两种模式
- **循环控制**：支持循环执行，避免循环套循环导致的死循环问题
- **条件控制**：支持条件判断，满足条件才执行相应步骤

#### 3. 断言验证规则

- **类型自适应**：根据实际值和期望值的类型自动选择合适的断言方法
- **多重断言**：单个步骤支持多个断言条件，所有断言都通过才算成功
- **断言失败处理**：断言失败时记录详细的错误信息和对比结果

#### 4. 资源管理规则

- **自动清理**：任务执行完成后自动清理本地测试数据和临时文件
- **截图管理**：测试过程中的截图按任务 ID 分目录存储，支持自动上传
- **日志管理**：按模块和时间分类记录日志，支持调试日志和错误日志

### 异常处理机制

#### 1. 连接异常处理

- **心跳重连**：心跳连接断开时自动重连，重连间隔 1 秒
- **API 重试**：平台接口调用失败时自动重试，最多重试 2 次
- **Token 刷新**：Token 过期时自动申请新的 Token 并重试请求

#### 2. 任务执行异常处理

- **进程隔离**：任务执行异常不影响其他任务的正常执行
- **异常捕获**：捕获并记录任务执行过程中的所有异常信息
- **资源释放**：异常发生时确保资源得到正确释放，避免资源泄漏

#### 3. 用例执行异常处理

- **断言异常**：断言失败时记录为测试失败，继续执行后续步骤（如果配置允许）
- **程序异常**：程序错误时记录为测试错误，根据配置决定是否继续执行
- **超时处理**：接口请求超时时抛出异常，避免长时间等待

#### 4. 数据异常处理

- **文件下载异常**：测试数据下载失败时记录错误并跳过任务执行
- **数据解析异常**：JSON 数据解析失败时使用默认值或抛出异常
- **参数渲染异常**：参数模板渲染失败时记录错误信息

## 数据设计

### 主要数据结构

#### 1. 任务数据结构

**Task（任务）：**

```python
{
    "taskId": str,              # 任务唯一标识
    "taskType": str,            # 任务类型（normal/debug）
    "maxThread": int,           # 最大并发线程数
    "reRun": bool,              # 是否支持失败重试
    "downloadUrl": str,         # 测试数据下载地址
    "debugData": dict,          # 调试模式数据（仅调试任务）
    "testCollectionList": [     # 测试集合列表
        {
            "collectionId": str,    # 测试集合ID
            "testCaseList": [       # 测试用例列表
                {
                    "caseId": str,      # 用例ID
                    "caseType": str,    # 用例类型（API/WEB/APP）
                    "index": int        # 执行索引
                }
            ]
        }
    ]
}
```

**TestCase（测试用例）：**

```python
{
    "driver": object,           # 测试驱动（WEB/APP）
    "session": object,          # 会话对象（API）
    "context": dict,            # 上下文变量
    "task_id": str,             # 任务ID
    "test_type": str,           # 测试类型
    "test_class": str,          # 测试类名
    "test_case": str,           # 测试用例名
    "test_data": str/dict       # 测试数据（文件路径或数据对象）
}
```

#### 2. 执行结果数据结构

**CaseResult（用例结果）：**

```python
{
    "taskId": str,              # 任务ID
    "collectionId": str,        # 测试集合ID
    "caseId": str,              # 用例ID
    "index": int,               # 执行索引
    "runIndex": int,            # 运行轮次
    "status": int,              # 执行状态（0:成功 1:失败 2:错误）

### 数据流转过程

#### 1. 任务数据流转

```
平台任务队列 → Api.fetch_task() → 任务队列 → Setting.task_analysis() 
→ 测试计划 → Run.run_test() → 测试套件 → Case.testEntrance() 
→ 具体测试模块执行
```

**详细流程：**

1. **任务获取**：Api 从平台获取待执行任务，包含任务基本信息和测试集合
2. **数据下载**：Setting 根据 downloadUrl 下载测试数据文件（ZIP 格式）
3. **数据解压**：将 ZIP 文件解压到本地数据目录，按任务 ID 分目录存储
4. **任务解析**：解析测试集合和用例列表，生成执行计划
5. **用例构建**：为每个用例创建 TestCase 对象，包含执行上下文
6. **线程分配**：根据测试集合分组，分配到不同线程执行

#### 2. 执行结果流转

```
测试步骤执行 → 事务结果 → Case.trans_list → 用例结果 → 结果队列 
→ Report.monitor_result() → 批量上传 → 平台结果存储
```

**详细流程：**

1. **事务记录**：每个测试步骤作为一个事务，记录执行状态、时长、日志等
2. **用例汇总**：用例执行完成后，汇总所有事务结果生成用例结果
3. **结果入队**：用例结果放入结果队列，等待批量上传
4. **批量上传**：Report 定期（3 秒间隔）批量上传结果到平台
5. **结果存储**：平台接收结果并存储到数据库

#### 3. 资源文件流转

```
测试执行 → 截图生成 → 本地存储 → Upload.set_upload() 
→ 平台上传 → 资源清理
```

**详细流程：**

1. **截图生成**：测试过程中自动截图，按时间戳命名
2. **本地存储**：截图文件存储到 image/{taskId}/ 目录
3. **自动上传**：Upload 定期扫描并上传截图文件
4. **资源清理**：任务完成后自动清理本地截图和数据文件

#### 4. 配置数据流转

```
config.ini → IniReader → Config → 各模块配置属性
```

**详细流程：**

1. **配置读取**：IniReader 读取 config.ini 配置文件
2. **配置解析**：Config 解析配置项并提供属性访问
3. **配置分发**：各模块通过 Config 实例获取所需配置
4. **动态更新**：支持运行时修改配置（如 Token 更新）

│   └── debug.log           # 调试日志
└── browser/                # 浏览器驱动目录
```

#### 2. 内存存储管理

**进程级存储：**

- `exec_processes`：当前执行的任务进程字典
- `message_queue`：消息队列，用于进程间通信
- `task_queue`：任务队列，存储待执行任务
- `case_result_queue`：结果队列，存储测试结果

**线程级存储：**

- `context`：测试上下文，存储参数变量
- `session`：HTTP 会话对象，复用连接
- `driver`：浏览器驱动对象
- `trans_list`：事务列表，记录执行步骤

#### 3. 数据生命周期管理

**任务数据生命周期：**

1. **创建**：任务开始时下载并解压测试数据
2. **使用**：测试执行过程中读取用例数据
3. **清理**：任务完成后自动删除本地数据

**截图数据生命周期：**

1. **生成**：测试过程中自动截图
2. **存储**：本地临时存储
3. **上传**：定期上传到平台
4. **清理**：上传成功后删除本地文件

**日志数据生命周期：**

1. **记录**：实时记录执行日志
2. **存储**：按模块分类存储
3. **轮转**：定期清理过期日志（可配置）

**配置数据生命周期：**

1. **加载**：系统启动时加载配置
2. **缓存**：内存中缓存配置对象
3. **更新**：支持运行时动态更新（如 Token）
4. **持久化**：配置变更时写回文件

#### 4. 数据安全和备份

**数据安全措施：**

- 敏感配置（如密钥）采用配置文件存储，不硬编码
- 临时数据及时清理，避免敏感信息泄露
- 网络传输采用 HTTPS 和 WSS 加密

**数据备份策略：**

- 重要日志文件定期备份
- 配置文件变更前自动备份
- 测试数据执行完成后及时清理，不做长期存储
